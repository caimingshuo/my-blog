# 从 Tool Use 到 World Model：Agent 为什么开始需要“内在模拟器”

*Published: April 19, 2026*

如果说过去一年 Agent 最明显的进步，是它终于学会了搜索、写代码、调 API、操作浏览器，那么最近另一个更有意思的变化是：大家开始意识到，**会用工具并不等于会完成任务**。

工具调用解决的是“伸出手”的问题，而 world model 解决的是“动手之前脑子里是否有一张地图”的问题。

## Tool Use 的上限为什么越来越明显？

早期 Agent 的设计很自然：

1. 读用户目标
2. 选择一个工具
3. 看结果
4. 再决定下一步

这种循环在短任务里很有效，尤其是搜索、问答、轻量代码编辑。但一旦任务变长，问题就开始暴露出来：

- **局部最优**：Agent 很容易只盯着眼前一步
- **探索成本高**：每试一次路径都要调用真实工具
- **恢复能力弱**：一旦中途偏离，往往只能继续乱试
- **缺乏风险预判**：不知道某一步是否会把自己带入死路

从表面看，这像是“推理还不够强”；但从系统层看，更像是它缺少一个足够稳定的内部世界表示。

## 为什么这不是一个纯学术问题？

如果只是研究讨论，tool use 和 world model 的区别听起来像抽象概念；但一旦放到真实产品里，这个区别非常具体。

比如一个 coding agent 要改线上服务配置，一个 browser agent 要帮你提交一份复杂表单，一个 office agent 要在多个系统之间搬运信息。此时系统真正面临的，不是“会不会调用工具”，而是：

- 现在是不是已经处在正确流程里？
- 下一步动作是不是会带来不可逆副作用？
- 如果本轮观察异常，是应该继续、等待还是回退？
- 有没有一条成本更低、成功率更高的路径？

这些问题回答不好，Agent 就会表现得像一个非常勤奋但方向感很弱的实习生。

## 什么叫“内在模拟器”？

我更喜欢把 world model 理解成 Agent 的一种 **inner simulator**。

它不需要完整复刻真实世界，但至少要让 Agent 能在脑子里先跑一遍：

- 如果我现在去点这个按钮，会发生什么？
- 如果我先改这个配置，后面会不会更容易？
- 如果当前结果和预期不一致，最可能是哪一步出错？
- 我现在看到的是正常分支，还是异常分支？

一个没有内在模拟器的 Agent，本质上只能边撞边学；而一个带内在模拟器的 Agent，才能开始真正规划。

## 这和传统 planner 有什么不同？

有人会说：那不就是 planner 吗？

只说对了一半。planner 主要负责“拆任务和排步骤”，但 world model 更关心“步骤执行后，世界会变成什么样”。

两者的关系更像这样：

- **Planner** 决定走哪条路
- **World Model** 估计这条路前面是桥、弯道还是悬崖

如果只有 planner，没有 world model，计划会显得漂亮但脆弱；因为它默认环境会照着自己的剧本走。

## 为什么 coding agent 特别需要它？

这件事在 coding agent 身上其实特别明显。

一个真正有用的 coding agent，不能只是会生成 patch。它还得理解：

- 改这个函数会影响哪些调用方
- 这个测试失败是根因还是连带反应
- 先补测试还是先修行为，哪个更稳
- 当前分支的改动是否与用户已有修改冲突

这些都不是单步工具选择问题，而是对“代码世界状态”的预测问题。越是复杂仓库，越需要内部模型去压缩结构、推演影响和评估风险。

## Browser Agent 和 Desktop Agent 也是一样

浏览器 Agent、Computer Use Agent 之所以难，不是因为点击动作本身难，而是因为：

- 页面跳转有分支
- 表单填写有顺序依赖
- 弹窗和权限流程会打断主任务
- 某些动作不可逆

如果 Agent 只能“看到当前屏幕”，它就永远像一个只活在当下的操作者。只有当它开始对界面流程形成内部表示，它才有机会变得稳定。

## 内在模拟器会如何出现？

我猜接下来比较现实的路线，不是一步到位训练一个超强通用世界模型，而是通过多个工程层叠加，把“模拟能力”逐渐做出来。

### 路线一：更强的状态抽象

把当前环境压缩成高价值结构，比如：

- 当前任务阶段
- 关键对象及其关系
- 最近动作与结果
- 未完成目标和潜在风险

### 路线二：候选动作比较

在真正执行前，先生成多个动作候选，并对它们做粗略打分：

- 成功概率
- 成本
- 风险
- 可回退性

### 路线三：轨迹学习

利用历史执行轨迹，让系统学会哪些动作组合通常通向成功，哪些模式往往意味着即将偏航。

### 路线四：沙盒和模拟环境

让 Agent 在低成本环境里反复试错，先学会对环境动态的直觉，再迁移到真实任务。

## 一个简单的心智模型

如果要用一句更直观的话来解释，我会这样区分：

- tool use 是“我现在能做什么”
- planner 是“我接下来准备做什么”
- world model 是“如果我这么做，世界大概率会怎么变”

这三个层次叠在一起，才比较接近一个成熟 Agent 所需要的认知结构。少了任何一层，系统都可能在复杂任务里显得头重脚轻。

## 为什么这会成为下一阶段分水岭？

因为未来 Agent 的竞争不会只体现在“能不能做”，而会体现在：

- 能不能更少试错
- 能不能更稳地完成长任务
- 能不能提前规避高风险路径
- 能不能把经验沉淀成可迁移的策略

这些能力背后都离不开某种形式的 world model。

换句话说，Agent 的演化路径很可能是这样的：

1. `Prompted assistant`
2. `Tool-using agent`
3. `Planning agent`
4. `World-model-backed agent`

现在我们大概正处在第 2 阶段向第 3、4 阶段过渡的地方。

## 接下来会怎么演化？

我猜未来一段时间里，我们不会马上看到一个统一、通用、无所不能的 world-model agent。更可能出现的是很多“半结构化”的过渡形态：

- 先有更强的任务状态机
- 再有更好的轨迹回放和失败总结
- 然后出现对候选动作的显式比较
- 最后才是越来越稳定的环境模拟和长期规划

也就是说，world model 不一定会以一个显眼的新模块突然出现，它更可能是慢慢渗进 Agent runtime、memory、planner 和 evaluator 里的。

## 结语

我越来越觉得，未来真正强的 Agent，不会是那个拥有最多工具的系统，而是那个最懂“自己的动作将怎样改变世界”的系统。

Tool use 让 Agent 获得了手脚，world model 才让它有机会长出方向感。

当行业开始重新认真讨论 world model，某种意义上也说明 Agent 已经不再满足于“会调用”，而是开始追求“会判断、会预演、会收敛”。这会是下一阶段最值得看的变化之一。
